# PicRecog 개발 지침 (GEMINI.md)

이 파일은 PicRecog 프로젝트의 아키텍처, 코딩 스타일, 그리고 작업 시 반드시 준수해야 할 규약들을 정의합니다. 모든 AI 에이전트와 개발자는 이 지침을 최우선으로 참고해야 합니다.

## 1. 프로젝트 개요
- **목적**: AI를 활용한 이미지 자동 분류 및 벡터 기반 유사 이미지 검색 시스템.
- **핵심 기술**: Python, Streamlit, SQLAlchemy, PostgreSQL (pgvector), LangGraph.

## 2. 백엔드 코딩 스타일 및 규약

### 자동 의존성 주입 (Dependency Injection)
- 데이터베이스 세션(`db: Session`)이 필요한 서비스 함수에는 `@transactional` 데코레이터를 사용합니다.
- 데코레이터 내에서 `inspect.signature(func)`를 사용하여 함수가 `db` 파라미터를 요구하는지 런타임에 확인하고 자동으로 주입합니다.
- **코드 공통화 원칙**: 중복 로직을 최소화하기 위해 내부 컨텍스트 매니저(`_transactional_session`)를 활용하여 동기/비동기 래퍼의 세션 생명주기를 통합 관리합니다.
- **원칙**: 개발자가 명시적으로 `db`를 전달하지 않아도 시스템이 자동으로 세션을 관리하도록 설계합니다.

### 이미지 처리 및 데이터 영속화
- **선 로드(Pre-loading) 원칙**: 워크플로우 시작 전 `Service` 레이어에서 이미지를 1회 로드 및 리사이징하여 바이트 데이터를 확보합니다. 이는 중복 I/O를 방지하고 모든 노드에서 동일한 데이터를 사용하게 합니다.
- **영속화 조건**: 분류 결과가 "no-class"이더라도 임베딩(벡터)이 정상적으로 생성되었다면 데이터베이스에 저장합니다. 단, 임베딩 생성 자체가 실패한 경우에는 저장을 생략하여 데이터 무결성을 유지합니다.

### 로깅 표준 (Logging Standard)
- **전면 금지**: 코드 내에서 직접적인 `print()` 함수 사용을 금지합니다.
- **표준 모듈**: 파이썬 표준 `logging` 모듈을 사용합니다.
- **설정**: `backend/system/config.py`의 `setup_logging()`을 통해 중앙 집중식으로 관리하며, `ui/app.py` 진입점에서 이를 초기화합니다.
- **레벨**: 개발 및 운영 환경에 따라 `LOG_LEVEL` 환경 변수를 통해 제어합니다.

### 트랜잭션 관리
- `@transactional` 데코레이터가 세션의 `commit`, `rollback`, `close` 생명주기를 전담합니다.
- **동적 지원**: 동기(sync) 및 비동기(async) 함수를 모두 지원하며, `inspect.iscoroutinefunction`을 통해 런타임에 호출 방식을 결정합니다.
- 비즈니스 로직 내부에서는 가급적 직접적인 commit/close를 피하고 데코레이터에 위임합니다.

### 비동기 처리 및 병렬 최적화 (Async Strategy)
- **이벤트 루프 보호**: 네트워크 바운드(API 호출)는 `async/await`를 사용하고, CPU 바운드(AI 모델 추론) 및 I/O 바운드(파일 로드)는 `asyncio.to_thread()`를 사용하여 이벤트 루프가 차단되는 것을 방지합니다.
- **클라이언트 비동기화**: 모든 AI 클라이언트는 `AsyncInferenceClient` 등 비동기 라이브러리를 우선적으로 활용하여 구현합니다.
- **워크플로우 싱글톤/재사용**: 폴더 처리 등 대량의 데이터 처리 시 워크플로우 인스턴스를 매번 생성하지 않고, 서비스 인스턴스 내에서 1회 생성(컴파일) 후 재사용하여 오버헤드를 최소화합니다.
- **Batch 엔진 활용**: 다수 이미지 처리 시 개별 `ainvoke` 대신 LangGraph의 `abatch()` 메서드를 사용하여 엔진 레벨의 최적화된 병렬 처리를 수행합니다.
- **전처리 병렬화**: 배치는 `asyncio.gather()`를 통해 이미지 로딩 등 전처리를 병렬로 실행한 후 워크플로우 배치 엔진에 공급합니다.

### 데이터 모델링
- 이미지 메타데이터는 `ImageRecord`에, 벡터 데이터는 `EmbeddingRecord`에 분리하여 저장합니다.
- 유사도 계산 시에는 `pgvector`의 `cosine_distance` 등을 활용하여 DB 레벨에서 처리합니다.

## 3. UI (Streamlit) 코딩 스타일 및 규약

### 사용자 경험 및 컨텍스트 유지
- **인라인 결과 표시**: 특정 항목에 대한 상세 정보나 유사 검색 결과를 보여줄 때, 별도의 페이지로 이동하지 않고 해당 항목 바로 하단(행 단위)에 결과 창을 렌더링합니다.
- **갤러리 연속성**: 검색 결과를 확인하는 중에도 기존의 목록(갤러리 등)이 사라지지 않도록 유지해야 합니다.

### 상태 관리
- `st.session_state`를 사용하여 현재 활성화된 검색 ID나 UI 상태를 추적합니다.
- 버튼 클릭 등 상태 변화가 발생하면 `st.rerun()`을 호출하여 UI를 즉시 갱신합니다.
- **금지 사항**: UI 렌더링 함수 중간에 `return`을 사용하여 하단 컴포넌트의 렌더링을 차단하는 행위는 지양합니다. (단, 명확한 페이지 전환 의도가 있는 경우는 제외)

### 실시간 비동기 UI 피드백 (Async UI Strategy)
- **비차단 UI**: 대량 이미지 처리 시 UI가 멈추지 않도록 `asyncio.create_task`를 사용하여 백그라운드 태스크를 실행합니다.
- **멀티 태스크 관리**: `st.session_state.active_tasks`를 통해 여러 작업을 동시에 추적하며, 각 작업은 독립적인 `asyncio.Queue`를 가집니다.
- **부분 렌더링**: `st.fragment`를 사용하여 전체 페이지 리런 없이 각 작업의 진행 상태바와 실시간 노드명(Progress)을 갱신합니다.

### 이벤트 및 로깅 분리 (Event vs Log)
- **원칙**: UI 피드백을 위한 '이벤트'와 시스템 추적을 위한 '로그'를 엄격히 분리합니다.
- **UIEventHandler**: `publish()` 메서드를 통해 `status`, `progress`, `result`, `error` 타입의 이벤트를 큐에 발행합니다. 노드명(`node`)과 파일 경로(`path`)를 포함하여 상세한 진행 상황을 전달합니다.
- **WorkflowLogger**: 표준 로깅(`log()`)과 워크플로우 상태 저장용 로그(`make_state_log()`)를 담당하며, `ClassificationState`의 기록용으로만 사용합니다.
